## Background — What an EDR Actually Is

**EDR (Endpoint Detection and Response)** is a security solution deployed as a lightweight agent on every endpoint in an organisation. The agent continuously collects telemetry — process creation events, file operations, registry modifications, network connections, command-line arguments, and user context — and streams it to a central console where:

1. Detections are generated by behavioural analytics, threat intelligence, and anomaly engines
2. Alerts are enriched with context (asset inventory, user identity, threat intel labels, MITRE ATT&CK technique IDs)
3. Analysts investigate using process chain visualisations, IOC tables, and timeline views
4. Response actions can be triggered — host isolation, process kill, file quarantine, network block

### EDR vs Traditional Antivirus — The Critical Distinction

| Dimension | Traditional Antivirus | EDR |
| --- | --- | --- |
| Detection model | Signature-based (known file hashes, patterns) | Behaviour-based + signatures + threat intel |
| Scope of visibility | Single file at the moment of execution | Full process chain, parent-child relationships, command lines, network behaviour |
| Response | Block / quarantine the file | Isolate host, kill process tree, roll back changes, hunt across estate |
| Detection of fileless attacks | Limited or none | Strong — sees PowerShell command lines, in-memory injection patterns |
| Investigation surface | Minimal log | Full timeline + process tree + IOC enrichment |
| Required for modern SOC | No | Yes |

The single most important difference: **AV sees a file. EDR sees a story.** When an Office document spawns CMD which spawns CURL which writes to disk and contacts an external domain, AV might only flag the final dropped file (if it matches a signature). EDR flags the *chain* — and the chain is what tells the analyst they're looking at initial access, not random user activity.

### Detection 1 — Initial Access via Malicious Office Document (DESKTOP-HR01)

#### What the EDR Showed

**Summary tab:** A macro-enabled Office document (`invoice.docm`) was opened using `WINWORD.EXE` by user `alice.thomas`. The document triggered `CMD.EXE`, which launched `cURL` to download a payload from an external domain. The file was saved to disk but never executed — behaviour consistent with malware staging tactics.

**Behaviour Summary:**
- Office macro launched CMD
- CMD triggered cURL to download payload
- File saved to Public folder
- Execution did not occur — potential staging

#### Q1 — Which tool was launched by CMD.exe to download the payload?

**Answer:** `CURL.exe`

**Where found:** Summary tab → Behavior Summary section.

**ATT&CK mapping:** T1059 (Command and Scripting Interpreter) + T1105 (Ingress Tool Transfer). The Office → CMD → CURL chain is one of the most documented Initial Access patterns in the wild — used historically by Emotet, Qakbot, IcedID, and dozens of commodity malware families.

#### Q2 — What is the absolute path to the downloaded malware?

**Answer:** `C:\Users\Public\install.exe`

**Where found:** IOC/Indicators tab.

#### IOC Table for This Detection

| Type | Value | Source | Action |
| --- | --- | --- | --- |
| File Path | `C:\Users\Public\install.exe` | Suspicious Binary | Quarantined |
| File Name | `install.exe` | Dropped Payload | Flagged |
| Domain | `ayebd.thm` | C2 Server | Blocked |
| IP Address | `1.161.138.92` | External Download | Logged |
| SHA256 | `9e107d9d372bb6826bd81d3542a419d6eaf1e3f5b94fc3b1f69413c5c30ef2e5` | Unknown Executable | Analyzed, Flagged |
| File Path | `C:\User\alice\Downloads\invoice.docm` | Suspicious Document | Quarantined |

This table is the analyst's whole evidence chain in one view. Every artefact (file, domain, IP, hash) is automatically captured and tagged with a status — the EDR has already taken several response actions (Quarantined, Blocked) before the analyst even opens the detection.

**Why the Public folder matters:** `C:\Users\Public\` is a low-permission, world-accessible directory often used by attackers because it doesn't require elevation, doesn't trigger user-folder monitoring rules, and survives most logged-in user contexts. Detection of execution-from-Public is a common EDR rule for this reason.

### Detection 2 — Credential Dumping via LSASS Memory Access (WIN-ENG-LAPTOP03)

#### What the EDR Showed

**Process Chain:** `syncsvc.exe` (spawned from `explorer.exe`, unsigned, running as user `haris.khan`) accessed the memory of `lsass.exe` and wrote the dump to disk.

**Behaviour:**
- Accessed memory of lsass.exe
- Wrote dump to disk

**Threat Intel:**
- Matches known credential dumping tool
- ATT&CK ID: **T1003.001** (OS Credential Dumping: LSASS Memory)

**Network Activity:**
- Attempted exfil to: `https://files-wetransfer.com/upload/session/ab12cd34ef56/dump_2025.dmp`
- Domain: `files-wetransfer.com` (typosquatted WeTransfer lookalike)
- IP: `100.42.28.64` on Port `80`
- **Status: Blocked by EDR**

**Registry Activity:**
- `HKLM\SYSTEM\CurrentControlSet\Services\syncsvc` (service installation)
- `HKCU\Software\Microsoft\Windows\CurrentVersion\Run\syncsvc` (user-level autorun for persistence)

#### Q3 — What is the absolute path to the suspicious syncsvc.exe?

**Answer:** `C:\Users\haris.khan\AppData\Local\Temp\syncsvc.exe`

**Where found:** IOC/Indicators tab.

#### Q4 — On which URL was the exfiltration attempt being made?

**Answer:** `https://files-wetransfer.com/upload/session/ab12cd34ef56/dump_2025.dmp`

**Where found:** Process Info tab → Network Activity → "Attempted exfil to".

#### Analysis

This is a textbook **credential dumping → exfiltration** chain:

1. Attacker dropped `syncsvc.exe` (the name itself is a defensive evasion technique — it mimics the legitimate Windows `syncsvc.dll` service host)
2. The binary opened a handle to LSASS to dump credentials in memory (Mimikatz-style behaviour)
3. The resulting dump file was uploaded to a typosquatted WeTransfer domain — leveraging a familiar brand to evade URL reputation filters
4. Persistence was set via both HKLM (service) and HKCU (autorun) — belt-and-braces persistence pattern

The EDR **blocked the exfiltration at the network layer** — the dump never left the host. Without EDR, this would have been a full credential breach with downstream lateral movement risk.

| Type | Value | Source | Action |
| --- | --- | --- | --- |
| File Path | `C:\Users\haris.khan\AppData\Local\Temp\syncsvc.exe` | Suspicious Binary | Flagged, Quarantined |
| IP Address | `100.42.28.64` | C2 / Exfil Attempt | Blocked by Firewall |
| Domain | `files-wetransfer.com` | Malicious Exfil Domain | Blocked |
| Registry Key | `HKCU\...\Run\syncsvc` | Persistence | Flagged |
| SHA256 | `7a1c5f2c2d451e9e27b2d3eec5a1989b7796f5c45a647889bf36c1f71d6fe8e0` | File Hash | Quarantined |

### Detection 3 — Execution from AppData Directory (DESKTOP-DEV01) — The Most Educational Case

#### What the EDR Showed

**Process Chain:** `explorer.exe` → `UpdateAgent.exe`

| Field | Value |
| --- | --- |
| Path | `C:\Users\daniel.richards\AppData\Roaming\UpdateAgent.exe` |
| Command Line | Same as Path (no arguments) |
| PID | 1372 |
| Parent Process | `explorer.exe` |
| Signed | **No** |
| Network Activity | Outbound to `10.10.20.5:8080` (HTTP) |
| Hash (SHA256) | `9f9f2e4a44d1a3f711e26c8ab60b392d34e2b36b012f...` |
| Start Time | 13:09:11 UTC |
| **Threat Intel** | **Known internal IT utility tool** |

#### Q5 — What was UpdateAgent.exe labelled by Threat Intel?

**Answer:** `Known internal IT utility tool`

**Where found:** Process Info tab → Threat Intel field.

#### Why This Case Is the Whole Point of EDR

Read the behavioural indicators in isolation and you would absolutely escalate this:

- ❌ Executable in **AppData\Roaming** — classic malware staging location
- ❌ **Unsigned** binary — no Authenticode trust
- ❌ Spawned by **explorer.exe** — common indicator of user-initiated launch of something they shouldn't have run
- ❌ **Outbound HTTP** to an internal but non-standard IP/port (`10.10.20.5:8080`)
- ❌ Filename `UpdateAgent.exe` — generic, masquerading-friendly name

By every behavioural rubric, this looks like commodity malware (think Lumma Stealer, RedLine, or basic infostealers — all of which routinely live in `AppData\Roaming` and beacon to non-standard internal-looking IPs after compromise).

**But the Threat Intel tag flips the verdict entirely.** `UpdateAgent.exe` is a legitimate internal IT utility. The "suspicious" behaviour is actually expected because internal tools are often deployed to AppData, signed loosely or not at all, and communicate with internal management servers on non-standard ports.

#### The Lesson This Case Teaches

This is exactly why EDR exists — and exactly why it matters for SOC analyst training:

1. **Behaviour alone is not a verdict.** A high-confidence behavioural signal *plus* asset/threat-intel context is what produces a correct verdict
2. **False positives often look identical to true positives at the telemetry layer.** Without enriched context, an analyst either over-escalates (drowning the SOC in noise — see the SOC Metrics room's lesson on FP-driven burnout) or under-escalates (missing real attacks)
3. **Threat Intel enrichment is the difference between SOC L1 and SOC L0.** A junior team without threat intel feeds escalates everything to L2. A team with threat intel triages confidently and reserves L2 escalation for the cases that genuinely warrant it
4. **This is exactly why workbooks built around decision diamonds matter** (see the Workbooks room) — *"Is the source IP listed in asset inventory? Does the binary match a known good utility?"* are the questions that turn behavioural alerts into correct verdicts

## Cross-Detection Lessons Carried Forward

1. **EDR is the SOC analyst's primary investigation surface.** Every other room in SOC Level 1 — alert triage, reporting, workbooks, metrics — assumes the analyst is operating *out of* an EDR (or SIEM that consumes EDR telemetry). Fluency here is the foundation
2. **The four tabs (Summary, Process Info, IOC/Indicators, Actions/Response) are the analyst's standard pivots.** Summary for the narrative; Process Info for the tree and command lines; IOC/Indicators for the atomic evidence; Actions/Response for what the EDR has already done and what's still required
3. **EDR automatically takes containment actions before the analyst sees the alert.** In the lab, the EDR had already Quarantined files, Blocked domains, and stopped exfiltration. The analyst's job isn't to *start* response — it's to *verify, complete, and document* it
4. **Process chains tell the story.** `explorer.exe → UpdateAgent.exe`, `WINWORD.EXE → CMD.EXE → CURL.EXE`, `explorer.exe → syncsvc.exe → lsass.exe handle` — each chain is a sentence describing what happened. Reading them is the core literacy of SOC L1 work
5. **Context flips verdicts.** This is the single biggest lesson of the room. Threat Intel enrichment, asset inventory lookups, user identity context — these are what separate behavioural noise from genuine incidents
6. **MITRE ATT&CK is embedded everywhere.** T1003.001 (LSASS Memory), T1059 (Command and Scripting Interpreter), T1105 (Ingress Tool Transfer), T1547 (Registry Run Keys) — the EDR maps detections to ATT&CK automatically, reinforcing the framework's role as the SOC's lingua franca
